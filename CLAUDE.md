## Three-Role Architecture

| Role | Machine | Claude | Function |
|------|---------|--------|----------|
| **Architect** | Mac (Gareth's laptop) | Claude Code in guide-build vault | Designs specs, writes chunks, manages docs, coordinates build |
| **Engineer** | guide-server (HP Z8 G4, Ubuntu 24.04) | Claude Code on bare metal | Executes chunks, writes and runs code, commits to guide-core |
| **Vault** | Mac Mini M2 Pro (pending migration to guide-server) | OpenClaw main agent | Live ops — serves team via Telegram/Slack, delivers briefs, observes channels |

**How they interact:**
- **Architect → Engineer:** CHUNK spec committed to vault. Engineer reads and executes directly.
- **Architect → Vault:** Workspace files updated (SOUL, AGENTS, TOOLS). Gateway restarted. Vault picks up new identity/rules.
- **Engineer → Vault:** Code deployed, gateway restarted. Vault operates new capabilities.
- **Vault → Gareth:** Surfaces what needs building (Engineer) or designing (Architect). Vault does not self-engineer.

**The Guide machine has full context:**
- This vault (synced) — all specs, briefs, CLAUDE.md files
- `guide-core` and `guide-engine` repos locally

The Three-Role split is about **role** (design / execution / operations), not access. All three can read everything. The Architect decides what to build; the Engineer decides how; the Vault operates it.

## Multi-Brand Model

Guide uses a hybrid architecture for three brands:

- **Shared agents** (Guide, Pipeline, Briefing, Scribe, Analyst, Finance) — serve all brands
- **Brand-specific agents** (SEO, Paid, HubSpot, Product × 3 brands) — created via agent factory
- **Cross-brand agents** (Apex, CapitalCore) — portfolio-level intelligence

Brand-specific agents are generated from templates using the agent factory (`guide-core/agent-factory/`).

## Master Roster: guide-roster.json

`Specs/guide-roster.json` is the single source of truth for all personal instances, team vaults, channel agents, and API keys. It includes deployment gates (boolean checklists) and status tracking.

- **Gareth maintains it** in this vault
- **Engineer copies it** to `~/guide-core/agent-factory/roster.json` on the Guide machine
- **`generate.sh personal {name}`** reads person config directly from roster.json
- **Before any personal instance work**, read the person's entry in roster.json — check their gates and status

Key sections: `persons` (identity, vaults, gates), `teamVaults` (source, gates), `channelAgents` (Slack bindings, gates), `apiKeys` (provisioned status, key locations).

## Known Pitfalls

### openclaw.json — validate schema before adding keys

**Never add keys to `openclaw.json` without first verifying they exist in the current schema.**

OpenClaw uses `additionalProperties: false` throughout its config schema. Any unrecognised key causes a hard startup failure — the gateway crash-loops with no graceful degradation and no warning at write time.

**Incident (2026-04-17):** Two keys were added to `channels.slack` (`dmOutboundAllowlist`, `dmPoliteList`) that don't exist in the current schema. Guide crash-looped for ~10 hours overnight before being diagnosed and fixed.

**Rule:** Before adding any new key, run:
```
openclaw config schema
```
Check the schema for the exact field name and its location in the hierarchy. If the field doesn't exist, don't add it — note it as a future feature request instead.
