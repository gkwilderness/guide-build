### Issues discovered during Nick launch

1. **Verbose/thinking output** — bot was spitting internal reasoning into Telegram. Default is verbose. Engineer fixed manually. Added to pre-flight checklist.
2. **errorPolicy not set** — empty-message events (stickers, media without caption) caused LLM calls with empty messages arrays → 400 errors → retry loop. Burns API credits, causes slowness. Same fix already existed on main bot's group chat but wasn't applied to Nick. Engineer fixed. Added to pre-flight checklist.
3. **sharedMounts path bug** — roster.json `sharedMounts` entries like `"guide-outputs/reports"` were being prefixed with `~/guide-shared/`, producing invalid paths. Fixed: all `sharedMounts` now use full relative-to-home paths. Template updated to just prepend `~/`.
4. **tools.exec.ask invalid value** — roster.json used `"on"` which is not schema-valid. Changed to `"always"` across all 3 occurrences.
5. **System files visible to user** — Nick could see and read SOUL.md, IDENTITY.md etc. Led to the filesystem redesign (see below).
6. **Team vaults writable** — Nick successfully wrote a test file to `~/guide-teams/exec/` which was supposed to be read-only. Confirmed: write restrictions are prompt-level only (AGENTS.md/TOOLS.md), not filesystem-enforced. This is the correct model on bare metal — team vaults are writable by the team, agents are told not to write via prompt.
7. **No directory listing capability** — agents without exec cannot run `ls`. They can only read files by exact path. Solution: INDEX.md convention for all shared directories.
8. **Boot context files not auto-copied** — roster.json `bootContext` lists files to read at session start but factory doesn't copy them into workspace. Decision: correct behaviour — agent reads them from mounted team vaults at runtime, not from workspace copies.

## Filesystem Architecture Redesign

### Problem

The original architecture (CHUNK-12) moved agent workspaces OUT of `~/.openclaw/` into `~/guide-vault/`, creating a custom directory structure that fought OpenClaw's native layout. Personal instance workspaces mixed system files and user content in the same directory.

### Approved Layout

Canonical reference: `Specs/guide-filesystem-layout.md`

**Principle:** OpenClaw workspaces (system) stay inside `~/.openclaw/`. User content (vaults) lives under `~/guide-vault/`.

```
~/.openclaw/                              ← OpenClaw's home
├── workspace/                            ← Main agent (system files)
├── workspace-personal-nick/              ← Nick agent (system files)
├── workspace-personal-{name}/            ← Other personal agents
├── workspace-{role}/                     ← Channel agents (data, seo, etc.)
└── ...

~/guide-vault/personal/{name}/            ← Personal vaults (read-write)
~/guide-vault/teams/{team}/               ← Team vaults (read-write)
~/guide-vault/shared/{category}/          ← Shared data (read-write)
~/guide-outputs/                          ← Agent outputs (append-only)
```

### Key decisions

- Agent workspaces go back to `~/.openclaw/workspace-*` (OpenClaw's native location)
- `~/guide-vault/` is for USER content, not system files
- `~/guide-teams/` and `~/guide-shared/` eliminated as top-level dirs — consolidated under `~/guide-vault/teams/` and `~/guide-vault/shared/`
- `exec` renamed to `exco`, `people` renamed to `hr`
- Shared data expanded with new subdirs: business, impact, camps, sales, countries, regions
- Team vaults and shared data are read-write for all agents (not read-only as originally designed)
- `guide-outputs/` stays append-only
- Engineer prompt written for the migration: `Prompts/PROMPT_Engineer-filesystem-migration.md`
